# Registration Data: What Changed (and What Didn't)

## The four data categories are identical across main and linkme

| Data | Purpose | Content |
|------|---------|---------|
| Call defs | R `.Call` routing | C wrapper fn pointer + name + arg count |
| R wrappers | Generated R code | String fragments (roxygen + function bodies) |
| ALTREP regs | ALTREP class init | Registration function pointers |
| Trait dispatch | Cross-package vtable lookup | (concrete_tag, trait_tag) → vtable ptr |

The proc macro generates the same C wrappers, same R wrapper text, same ALTREP classes,
same vtables on both branches. The data is byte-for-byte identical.

## What changed: who collects

| | main (miniextendr_module!) | linkme branch |
|---|---|---|
| **Collection** | User lists items by name in macro | Linker gathers `#[distributed_slice]` entries |
| **Registration fn** | Module macro generates `R_init_{pkg}_miniextendr()` with explicit refs | `miniextendr_register_routines()` reads slices |
| **R wrappers** | Module macro generates `document.rs.in` template collecting named consts | `write_r_wrappers_to_file()` reads `MX_R_WRAPPERS` slice |
| **Trait dispatch** | Module macro generates per-type `if/else` query fn | `universal_query()` scans `MX_TRAIT_DISPATCH` slice |

## The archive extraction problem is a collection problem

The fundamental issue: the linker only extracts `.o` files from a `.a` archive when something
references a symbol inside them.

- **On main**: `miniextendr_module!` generated a big function that explicitly referenced every
  C wrapper by name (`C_add`, `C_Counter__new`, etc.). This created the references the linker
  needed. No force_load required.

- **On linkme**: the C wrappers register themselves into linker sections. Nothing explicitly
  references them. So the linker drops them unless forced (`-Wl,-force_load`).

linkme moved the collection from "explicit references in user code" to "linker sections" —
but linker sections in unextracted `.o` files are invisible. force_load compensates by
extracting everything unconditionally.

## Three collection strategies

### A. User-maintained list (main branch, deleted)

```rust
miniextendr_module! {
    mod mypackage;
    fn add;         // ← user must list every item
    fn multiply;
    impl Counter;
    use helpers;
}
```

- Pro: explicit references → linker extracts correctly
- Con: manual maintenance, 220+ lines, lint crate exists to catch mismatches

### B. Linker sections (linkme branch, current)

```rust
#[miniextendr]   // ← emits #[distributed_slice] entry automatically
pub fn add(a: f64, b: f64) -> f64 { a + b }
```

- Pro: zero user boilerplate
- Con: requires force_load for staticlib→.so (linker doesn't extract unreferenced .o files)
- Con: document binary needs anchor function (link_registrations) for rlib case

### C. Build-time code generation (not yet implemented)

build.rs scans source files (miniextendr-lint already does this), predicts C symbol names,
and generates explicit references + registration code:

```rust
// AUTO-GENERATED by build.rs
extern "C" { fn C_add(); fn C_multiply(); fn C_Counter__new(); }

#[unsafe(no_mangle)]
pub unsafe extern "C-unwind" fn miniextendr_register_routines(dll: *mut DllInfo) {
    // ... collect from named consts or linkme slices
}
```

- Pro: no force_load (explicit references pull .o files)
- Pro: no user maintenance (build.rs generates automatically)
- Con: build.rs must predict exact C symbol names matching proc macro output
- Con: one `include!` line in user crate (for the generated file)

### Hybrid: C for staticlib, linkme for cdylib

Use approach C for the R package .so (staticlib, no force_load).
Use linkme for document generation (cdylib, all symbols present naturally).

build.rs generates the references → staticlib links correctly.
cdylib loads in R → reads linkme slices → writes R wrappers.

User crate has: one `include!` line, `crate-type = ["staticlib"]`, no document.rs, no `[[bin]]`.

## What build.rs already knows

miniextendr-lint's source scanner (`crate_index.rs`) already detects:

- Every `#[miniextendr]` function (name, arg count, visibility, cfg gates)
- Every `#[miniextendr]` impl block (type name, methods, labels, class system)
- Every `#[miniextendr]` trait impl (trait name, concrete type, methods)
- Every `#[derive(Altrep)]` / `#[derive(Vctrs)]` type
- Module tree structure and cfg feature gates

The missing piece: computing C wrapper symbol names from this data. The naming convention
is deterministic (e.g., `fn add` → `C_add`, `Type::method` → `C_Type__method`) but must
exactly match what the proc macro generates.
