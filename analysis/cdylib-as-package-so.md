# cdylib for Document Generation + staticlib for R Package

## Constraint

**R CMD INSTALL requires staticlib.** R's build process compiles `.c` files in `src/` to
`.o` files, then links them with `PKG_LIBS` to produce the final `$(SHLIB)`. It expects to
control the final link step. You cannot hand R a pre-built `.so` — it must link the staticlib
itself.

This means:
- **staticlib** remains the crate-type for the R package .so
- **cdylib** is built separately, used only for document generation (R wrapper writing)
- Archive extraction (getting linkme entries into the .so) still needs a solution

## Two-artifact architecture

```
cargo build --lib                       → staticlib (.a)  → linked by R into pkg.so
cargo rustc --crate-type cdylib --lib   → cdylib (.dylib) → loaded by R for wrappers
```

The staticlib goes through R's build process. The cdylib is a build-time tool only — never
shipped, never installed.

## What the cdylib solves

The cdylib replaces the document binary entirely:

| Current (linkme branch) | With cdylib |
|------------------------|-------------|
| `document.rs` binary in user crate | R loads cdylib directly |
| `[[bin]]` in Cargo.toml | Not needed |
| `crate-type = ["staticlib", "lib"]` | `crate-type = ["staticlib"]` |
| `miniextendr_link_registrations()` in lib.rs | Not needed |
| `generate_link_registrations()` in lint | Not needed |
| `include!` in lib.rs | Not needed (for document — see below) |

cdylib includes all symbols by design — linkme entries, C wrappers, everything.
No anchor function needed. R loads it, reads the slices, writes the file.

### Document generation flow

```r
# During devtools::document or equivalent
cdylib_path <- "path/to/target/release/libmypkg.dylib"
lib <- dyn.load(cdylib_path)
.Call(getNativeSymbolInfo("miniextendr_write_wrappers", lib), "R/mypkg-wrappers.R")
dyn.unload(lib)
# Then roxygen2 reads the wrappers → generates NAMESPACE
```

Requires `miniextendr_write_wrappers` as `#[no_mangle] extern "C"` in miniextendr-api:

```rust
#[no_mangle]
pub unsafe extern "C" fn miniextendr_write_wrappers(path_sexp: SEXP) -> SEXP {
    let path = CStr::from_ptr(R_CHAR(STRING_ELT(path_sexp, 0)))
        .to_str()
        .expect("path must be UTF-8");
    write_r_wrappers_to_file(path);
    R_NilValue
}
```

## What the cdylib does NOT solve

The staticlib→.so link still drops unreferenced `.o` files from the archive. The linkme
`#[distributed_slice]` entries live in those `.o` files. Without something forcing their
extraction, they're lost.

**This is the archive extraction problem, and it requires a separate solution.**

### Option A: Keep force_load (current approach)

```makefile
PKG_LIBS = -Wl,-force_load,$(CARGO_AR)      # macOS
PKG_LIBS = -Wl,--whole-archive $(CARGO_AR) -Wl,--no-whole-archive  # Linux
```

Simplest. Works. Platform-specific.

### Option B: build.rs generates explicit references

build.rs (miniextendr-lint) already scans every `#[miniextendr]` item. It can generate
`extern "C"` declarations that create undefined symbol references, forcing the linker to
extract those `.o` files:

```rust
// AUTO-GENERATED by build.rs in OUT_DIR
// These extern declarations force the linker to extract the .o files
// containing these C wrapper functions. Once extracted, the linkme
// distributed_slice entries in those .o files are also included.

extern "C" {
    fn C_add();
    fn C_multiply();
    fn C_Counter__new();
    fn C_Counter__get_count();
    // ... every C wrapper symbol
}

// Array reference prevents dead-code elimination
#[used]
#[doc(hidden)]
static __MINIEXTENDR_FORCE_LINK: [unsafe extern "C" fn(); 4] = [
    C_add,
    C_multiply,
    C_Counter__new,
    C_Counter__get_count,
];
```

User crate includes this:
```rust
include!(concat!(env!("OUT_DIR"), "/miniextendr_force_link.rs"));
```

**Pro:** No force_load. Standard linking — only extracts what's referenced.
**Con:** One `include!` line. build.rs must predict exact C symbol names.

The C symbol naming convention is deterministic:
- `fn foo` → `C_foo`
- `Type::method` → `C_Type__method`
- `Type(label)::method` → `C_Type_label__method`
- `impl Trait for Type` method → `C_Trait_Type__method`

miniextendr-lint already knows every item. It just needs to compute names using the same
rules as the proc macro.

### Option C: miniextendr_init! generates the references

Instead of build.rs, the `miniextendr_init!` proc macro could expand to include the
force-link array. But proc macros don't have global state — they can't enumerate all
`#[miniextendr]` items across the crate. Only build.rs (which scans files) can do this.

So Option B (build.rs) is the viable path for eliminating force_load.

## Moving entrypoint.c and mx_abi.c to Rust

Independent of the cdylib/staticlib question, both C files can move to Rust. This is
valuable regardless — it reduces the number of files in `src/` and eliminates configure
template substitution for these files.

### entrypoint.c → Rust

`entrypoint.c.in` is a thin dispatcher calling Rust `extern "C"` functions:

```c
void R_init_@PACKAGE_NAME@(DllInfo *dll) {
    miniextendr_panic_hook();          // Rust
    miniextendr_runtime_init();        // Rust
    miniextendr_assert_utf8_locale();  // Rust
    miniextendr_set_altrep_pkg_name("@PACKAGE_NAME@");  // Rust
    miniextendr_init_vctrs();          // Rust
    mx_abi_register();                 // C (mx_abi.c) — but can be Rust
    miniextendr_register_routines(dll); // Rust
    R_useDynamicSymbols(dll, FALSE);   // R API — available via ffi.rs
    R_forceSymbols(dll, TRUE);         // R API — available via ffi.rs
}
```

Replace with `miniextendr_init!` proc macro + `package_init()` in miniextendr-api:

```rust
// User crate
miniextendr_init!(mypkg);

// Expands to:
#[unsafe(no_mangle)]
pub unsafe extern "C-unwind" fn R_init_mypkg(
    dll: *mut ::miniextendr_api::ffi::DllInfo,
) {
    ::miniextendr_api::package_init(dll, c"mypkg");
}
```

```rust
// miniextendr-api
pub unsafe fn package_init(dll: *mut DllInfo, pkg_name: &CStr) {
    miniextendr_panic_hook();
    miniextendr_runtime_init();
    miniextendr_assert_utf8_locale();
    miniextendr_set_altrep_pkg_name(pkg_name.as_ptr());
    let _ = miniextendr_init_vctrs();
    mx_abi_register(pkg_name);
    miniextendr_register_routines(dll);
    R_useDynamicSymbols(dll, 0);
    R_forceSymbols(dll, 1);
}
```

The `@PACKAGE_NAME@` configure substitution becomes a compile-time string literal passed
by the proc macro. No more `.in` template.

### mx_abi.c → Rust

`mx_abi.c` (127 lines) implements:

| Function | What it does | R API used |
|----------|-------------|------------|
| `init_tag()` | `Rf_install("miniextendr::mx_erased")` + `R_PreserveObject` | Yes, available in ffi.rs |
| `mx_externalptr_finalizer()` | `R_ExternalPtrAddr` + `base->drop()` + `R_ClearExternalPtr` | Yes |
| `mx_wrap()` | `R_MakeExternalPtr` + `R_RegisterCFinalizerEx` | Yes |
| `mx_get()` | `TYPEOF` + `R_ExternalPtrTag` + `R_ExternalPtrAddr` | Yes |
| `mx_query()` | `mx_get` + `base->query()` | Indirect |
| `mx_abi_register()` | `R_RegisterCCallable` × 3 | Yes |

All use public R API that miniextendr-api already has FFI bindings for. The Rust
implementation would live in `miniextendr-api/src/mx_abi.rs`. Functions stay
`#[no_mangle] pub unsafe extern "C"` for `R_GetCCallable` compatibility.

`mx_abi.h` (inst/include/) stays unchanged — consumer packages include it for type defs.

## What the user crate looks like

### With force_load (Option A — simplest)

```toml
# Cargo.toml
[lib]
crate-type = ["staticlib"]
```

```rust
// lib.rs — no include!, no document.rs, no [[bin]]
use miniextendr_api::miniextendr;

#[miniextendr]
pub fn add(a: f64, b: f64) -> f64 { a + b }

miniextendr_init!(mypkg);
```

```rust
// build.rs
fn main() { miniextendr_lint::build_script(); }
```

Makevars still has force_load. But user crate is clean.

### Without force_load (Option B — cleanest)

Same as above, plus one `include!` line:

```rust
// lib.rs
use miniextendr_api::miniextendr;

#[miniextendr]
pub fn add(a: f64, b: f64) -> f64 { a + b }

miniextendr_init!(mypkg);

// Force linker to extract .o files containing #[miniextendr] C wrappers
include!(concat!(env!("OUT_DIR"), "/miniextendr_force_link.rs"));
```

Makevars uses standard `PKG_LIBS = -L$(CARGO_LIBDIR) -l$(CARGO_STATICLIB_NAME)`.
No platform-specific force_load flags.

## Platform considerations

### cdylib for document generation

| Platform | cargo cdylib output | R loads via |
|----------|-------------------|-------------|
| macOS | `lib{name}.dylib` | `dyn.load()` |
| Linux | `lib{name}.so` | `dyn.load()` |
| Windows | `{name}.dll` | `dyn.load()` |

miniextendr-api's build.rs already links against libR, so the cdylib links correctly.

### staticlib for R package

Same as current. R's build system links `PKG_LIBS` into `$(SHLIB)`.

### Building both

`cargo rustc --crate-type cdylib` (stable since Rust 1.64) overrides crate-type without
modifying Cargo.toml. Shares incremental compilation artifacts with the staticlib build.

```bash
# Makevars or justfile
cargo build --lib --profile release               # staticlib (from Cargo.toml)
cargo rustc --crate-type cdylib --lib --profile release  # cdylib (override)
```

### Windows win.def

With entrypoint.c gone, `R_init_*` is in the staticlib (generated by `miniextendr_init!`).
R's linker needs to see it as an export. Options:
- Keep win.def.in (just the one-line `EXPORTS R_init_@PACKAGE_NAME@`)
- Or use `#[link(kind = "raw-dylib")]` / linker flags in build.rs

Keeping win.def.in for now is simplest.

## Migration path

### Phase 1: Move mx_abi.c to Rust (independent, low risk)
- Implement mx_wrap/mx_get/mx_query/mx_abi_register in miniextendr-api
- Keep entrypoint.c calling the Rust versions
- Delete mx_abi.c.in from all packages/templates

### Phase 2: Move R_init to Rust (independent, low risk)
- Implement `miniextendr_init!` proc macro
- Implement `package_init()` in miniextendr-api
- Replace entrypoint.c with `miniextendr_init!(pkg)` in lib.rs
- Delete entrypoint.c.in from all packages/templates

### Phase 3: cdylib for document generation
- Add `miniextendr_write_wrappers` SEXP wrapper in miniextendr-api
- Update devtools-document to: build cdylib, load from R, call writer
- Delete document.rs, `[[bin]]`, `crate-type = "lib"`
- Delete `miniextendr_link_registrations` and `generate_link_registrations`

### Phase 4 (optional): Eliminate force_load
- Extend build.rs to generate force_link.rs with explicit `extern "C"` references
- Add `include!` for force_link.rs in lib.rs
- Replace force_load in Makevars with standard `-L -l` linking
- Codify C symbol naming convention shared between proc macro and build.rs

### Phase 5: Update templates + tests
- Update minirextendr templates
- Update producer.pkg / consumer.pkg / model_project
- Run full test suite
