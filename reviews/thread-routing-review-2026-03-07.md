# Review: Worker/Main-Thread/Rayon Routing

## Findings

1. [P1] `run_on_worker` can deadlock on re-entry.

`run_on_worker_inner()` always submits work through the single global `JOB_TX`, and that queue is a capacity-0 rendezvous channel with exactly one receiver on the dedicated worker thread ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs:294](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs#L294), [/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs:339](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs#L339), [/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs:529](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs#L529)). If code that is already running as a worker job calls `run_on_worker`/`run_on_worker_result` again, or if a `with_r_thread` callback on the main thread tries to enqueue another worker job while the current worker is blocked waiting for that callback to finish, `job_tx.send(job)` cannot complete until the current job returns. There is no re-entry guard or fallback-to-inline path, so the result is a hard deadlock rather than a clear panic.

2. [P1] The non-`worker-thread` stub turns `with_r_thread` into wrong-thread inline execution.

When `worker-thread` is disabled, `with_r_thread` stops routing and simply runs the closure on the caller thread after an init check ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs:636](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs#L636)). At the same time, `#[r_ffi_checked]` still expands every non-variadic/non-`link_name` FFI wrapper to call `with_r_thread` ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs:2533](/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs#L2533), [/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs:2626](/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs#L2626)), and public APIs such as `ExternalPtr::new()` rely on that contract for cross-thread safety ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/externalptr.rs:327](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/externalptr.rs#L327)). In that feature configuration, a background thread calling a checked FFI wrapper or `ExternalPtr::new()` will execute the R API directly on the wrong thread instead of routing or panicking. This is especially easy to miss because `rayon` does not imply `worker-thread`.

3. [P2] The documented contract for pointer-returning `#[r_ffi_checked]` wrappers no longer matches the generated code.

Both the macro docs and the worker docs still say pointer-returning APIs like `INTEGER`, `REAL`, and `DATAPTR` must panic off the main thread ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs:2492](/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs#L2492), [/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs:39](/Users/elea/Documents/GitHub/miniextendr/miniextendr-api/src/worker.rs#L39)). The macro no longer implements that distinction: every ordinary return type is wrapped in `Sendable` and routed through `with_r_thread`, including raw pointers ([/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs:2634](/Users/elea/Documents/GitHub/miniextendr/miniextendr-macros/src/lib.rs#L2634)). The repo's own wrong-thread test comments are still written against the old invariant ([/Users/elea/Documents/GitHub/miniextendr/rpkg/src/rust/worker_tests.rs:400](/Users/elea/Documents/GitHub/miniextendr/rpkg/src/rust/worker_tests.rs#L400), [/Users/elea/Documents/GitHub/miniextendr/rpkg/tests/testthat/test-worker.R:141](/Users/elea/Documents/GitHub/miniextendr/rpkg/tests/testthat/test-worker.R#L141)). This is a correctness issue for maintainers: anyone reasoning from the docs will infer the wrong safety and liveness constraints around worker/Rayon handoff.
