## 1. Architecture Coherence

The module split is sensible and the responsibility boundaries are mostly clean:

| Module | Responsibility |
|--------|----------------|
| `main.rs` | CLI parsing, top-level orchestration (steps 0–14) |
| `vendor.rs` | Core vendoring: `cargo vendor` shell-out, archive extraction, path rewriting, checksum clearing, freeze, compress |
| `package.rs` | `cargo package` shell-out, workspace-inheritance patch, version injection for path-only deps |
| `metadata.rs` | `cargo_metadata` wrapper, partition into local/external packages |
| `cache.rs` | Content-hash cache to skip unchanged re-vendors |
| `strip.rs` | Dir/TOML section removal |
| `verify.rs` | Lock ↔ vendor ↔ tarball agreement checks |

**One muddled boundary**: `generate_cargo_config` (vendor.rs:612) takes `_local_pkgs: &[LocalPackage]` (prefixed with `_` — it's unused). The docstring says it also writes to `<vendor_dir>/../src/rust/.cargo/config.toml` "if that path exists", but the implementation does not actually do that — it only writes to `vendor/.cargo-config.toml`. The doc comment is wrong, and the parameter is dead weight. This is a minor but misleading inconsistency.

```rust
// vendor.rs:612–653 — _local_pkgs never read; docstring describes behavior
// that does not exist in the code
pub fn generate_cargo_config(
    manifest_path: &Path,
    vendor_dir: &Path,
    _local_pkgs: &[LocalPackage],  // dead parameter
) -> Result<String> {
    ...
    // Write to vendor dir for reference
    let config_path = vendor_dir.join(".cargo-config.toml");  // only writes here
    std::fs::write(&config_path, &config)?;
```

The generated config is returned as a string but the caller in main.rs only logs it at debug verbosity and discards the return value — the `.cargo/config.toml` that Makevars.in actually uses is generated by `configure.ac` (from `cargo-config.toml.in`). The `.cargo-config.toml` in vendor/ is reference-only. This should be documented or the dead path dropped entirely.

**Coupling assessment**: `main.rs` → `vendor.rs` is the only non-trivial coupling and it is clean. `package.rs` and `metadata.rs` are consumed only by `main.rs` and transitively through `vendor.rs`'s `run_cargo_vendor`. No circular coupling detected.

## 2. Correctness vs. Upstream (`cargo/ops/vendor.rs`)

### Divergences that are required for CRAN

| Behavior | Upstream cargo-vendor | cargo-revendor |
|---|---|---|
| Workspace/path deps | Skips (skips `is_path()` sources) | Explicitly packages via `cargo package` + fallback copy |
| Checksums | Writes real SHA-256 per-file checksums | Clears all checksums to `{"files":{}}` |
| `[patch.*]` sections | Preserved | Stripped and re-added pointing at vendor/ (--freeze) |
| `.cargo/config.toml` | Printed to stdout for user to add manually | Written to vendor/.cargo-config.toml (reference only) |
| Tarball compression | Not supported | `--compress` with xattr suppression |
| Workspace inheritance | Resolved via `cargo package` | Manual regex in `resolve_workspace_inheritance` (fallback path) |

All of these divergences are justified by CRAN constraints. The real-checksum behavior would cause `cargo build --locked` to fail against the cleared checksums.

### Divergences that are potential bugs or reinventions

**2a. Git source URL extraction is line-based regex, not structural (vendor.rs:625–641)**

```rust
// vendor.rs:628–634
for line in manifest_content.lines() {
    if let Some(start) = line.find("git = \"https://") {
        let url_start = start + 7; // skip `git = "`
```

The magic constant `7` skips `git = "` (exactly 7 characters: `g`,`i`,`t`,` `,`=`,` `,`"`). This only matches `git = "https://…"` not `git="https://…"` (no spaces). More critically it does not match `http://` git deps (rare but valid). Upstream uses `source_id.is_git()` from the resolved metadata, which is structurally correct. cargo-revendor's fallback produces a `.cargo-config.toml` that may miss git source replacements for deps declared with non-canonical spacing.

**2b. `--freeze` only rewrites deps for `local_pkgs`, not all git deps in Cargo.toml (vendor.rs:709–718)**

```rust
// vendor.rs:709–718
for section in &["dependencies", "build-dependencies"] {
    if let Some(table) = doc.get_mut(section).and_then(|v| v.as_table_mut()) {
        for pkg in local_pkgs.iter() {
            if let Some(dep) = table.get_mut(&pkg.name) {
                let dir_name = vendor_dir_name_for_pkg(...);
                rewrite_dep_to_vendor(dep, &dir_name, &vendor_rel);
```

`freeze_manifest` only rewrites dependencies whose name appears in `local_pkgs`. A Cargo.toml that declares a git dep on a crate that is NOT a local workspace member (e.g. `foo = { git = "https://github.com/…" }` for some external repo) will NOT have its dep rewritten to a vendor path. Instead, `freeze` relies on `[patch.crates-io]` entries for the external git crates. This works if the crate resolves via crates-io semantics, but breaks for git-only crates not published to crates.io: the `[patch.crates-io]` mechanism won't help because the dep is already a `git =` dep, not a crates-io dep.

Upstream resolves this differently: it uses source replacement in `.cargo/config.toml` which redirects ALL git sources. cargo-revendor's approach of writing source replacement into `.cargo-config.toml` IS correct for the CRAN case (the template configure.ac applies it), but the `--freeze` code does not actually write the necessary git source replacement entries into the manifest-side config, relying on a separate mechanism.

**2c. Duplicate version detection is absent**

Upstream (vendor.rs:188–203 in cargo) explicitly errors on `found duplicate version of package … vendored from two sources`. cargo-revendor has no equivalent. With `--sync` and conflicting sources, a silent last-write-wins could occur if two workspaces pull the same crate name+version from different git sources.

**2d. `dev-dependencies` not stripped from path-dep rewriting**

`rewrite_local_path_deps` (vendor.rs:256) also rewrites `dev-dependencies`, but `strip_vendor_path_deps` (vendor.rs:181) only strips from `dependencies` and `build-dependencies`. If a vendored crate had a dev-dep with a relative path dep (`path = "../sibling"`), it would be rewritten by `rewrite_local_path_deps` but not previously stripped by `strip_vendor_path_deps`. The asymmetry is harmless in practice (dev deps are typically stripped entirely by `--strip-all`) but confusing.

**2e. Workspace Cargo.toml mutation is not panic-safe**

In both `vendor.rs:run_cargo_vendor` (lines 44–71) and `package.rs:package_local_crates` (lines 60–91), the pattern is:
1. Read workspace Cargo.toml
2. Write patched version
3. Run cargo command
4. Write original back

If step 3 panics or the process is killed (SIGINT), the workspace Cargo.toml is left dirty with injected `[patch.crates-io]` entries. Upstream cargo never mutates the workspace manifest — it uses a resolved in-memory workspace representation. A `scopeguard`-style RAII restore would be safer:

```rust
// Risky pattern — vendor.rs:56–71
std::fs::write(&ws_manifest, format!("{}{}", ws_original, patch))?;
let output = cmd.output()...;
std::fs::write(&ws_manifest, &ws_original)?;  // not reached on panic!
```

The same issue affects `package.rs` lines 62–91. Both `package_local_crates` and `run_cargo_vendor` share this mutation+restore pattern without protection. In practice a `cargo vendor` invocation takes seconds, but a Ctrl-C during a slow crates.io download is not far-fetched.

## 3. Freeze Semantics: Does `--freeze` Actually Work Offline?

### When it works

The happy path covers the main miniextendr use case: R package with `git = "https://github.com/CGMossa/miniextendr"` deps for its workspace crates. `--freeze`:
1. Strips `[patch.*]` sections that reference the git URL
2. Adds `[patch.crates-io]` pointing local crates at vendor paths
3. Copies the vendor-stripped Cargo.lock

With configure.ac appending git source replacements to `.cargo/config.toml`, the full CRAN build works offline.

### Edge cases where freeze can break

**3a. Git dep on a non-crates-io crate (external git library not in local_pkgs)**

If `Cargo.toml` contains:
```toml
[dependencies]
some-external = { git = "https://github.com/thirdparty/some-external" }
```
and `some-external` is not a miniextendr workspace crate (not in `local_pkgs`), then:
- `freeze_manifest` does NOT rewrite this dep to a vendor path
- The crate does land in `vendor/` (cargo vendor vendors it)
- But the dependency declaration still says `git = "https://..."` 
- Offline build fails unless `.cargo/config.toml` has the right `[source."git+…"]` replacement

The git URL scanning in `generate_cargo_config` (the regex-based approach described in §2a) is supposed to catch this. But the generated config goes to `vendor/.cargo-config.toml` (reference only) — it's not written to `src/rust/.cargo/config.toml`. The actual config written by configure.ac's `AC_CONFIG_COMMANDS([dev-cargo-config])` scans git URLs from the Cargo.toml at configure time (line 328 in configure.ac), which also uses a sed-based regex. Both are fragile but at least configure.ac applies the result to the actual build config.

**3b. `dev-dependencies` git deps not frozen**

`freeze_manifest` iterates only `["dependencies", "build-dependencies"]` (vendor.rs:709), not `"dev-dependencies"`. A git dep in `[dev-dependencies]` would not be rewritten. This is probably OK for CRAN (dev deps are stripped), but the documentation doesn't call this out.

**3c. `regenerate_lockfile` copies vendor/Cargo.lock which itself was stripped of checksums**

`strip_lock_checksums` (vendor.rs:659) strips `checksum = ` lines from Cargo.lock and copies to vendor/Cargo.lock. `regenerate_lockfile` then copies that back to the manifest's Cargo.lock. This means the project's Cargo.lock permanently loses checksums after `--freeze`. That's intentional (vendored sources have empty checksums), but is not reversible without a fresh `cargo update`. The comment on line 880–884 explains the reasoning correctly.

## 4. Test Coverage Audit

### Critical gap: test harness broken by default versioned-dirs layout

**This is the most significant test issue found.**

Since PR #239 merged (versioned-dirs default-on), every `#[ignore]` integration test that calls `assert_vendor_has(&vendor, "cfg-if")`, `read_vendor_toml(&vendor, "cfg-if")`, or `assert_empty_checksum(&vendor, "cfg-if")` will fail at runtime. Affected helpers in `tests/common/mod.rs`:

```rust
    let cksum = vendor.join(name).join(".cargo-checksum.json");  // same issue
```

```rust
}
```

## 5. Template Correctness

### `Cargo.toml.tmpl` — git deps, no versioned-dir assumptions

The template correctly uses git deps for miniextendr crates (not path deps pointing at vendor/):
```toml
# Cargo.toml.tmpl:20–23
[dependencies]
miniextendr-api = { git = "https://github.com/CGMossa/miniextendr" }
[build-dependencies]
miniextendr-lint = { git = "https://github.com/CGMossa/miniextendr" }
```

This is the correct shape: `--freeze` during CRAN prep rewrites these git deps to vendor paths, and configure.ac appends source replacement entries. No versioned-dir assumptions are hardcoded in Cargo.toml.tmpl.

### `cargo-config.toml.in` — uses `@VENDOR_OUT_CARGO@` substitution

```toml
# cargo-config.toml.in:8–9
[source.vendored-sources]
directory = "@VENDOR_OUT_CARGO@"
```

This is `autoconf`-substituted at configure time to an absolute path. No hardcoded flat vendor path. Correct.

### `configure.ac` — `cargo revendor` invocation does NOT pass `--freeze`

The `cargo-vendor` AC_CONFIG_COMMANDS block (lines 436–442) invokes:
```sh
cargo revendor \
  --manifest-path src/rust/Cargo.toml \
  --output "$VENDOR_OUT" \
  --strip-all \
  --source-marker \
  $_source_root -v
```

**`--freeze` is intentionally absent here** — dev-mode vendoring should NOT freeze the manifest, since that would replace git deps with vendor paths and break subsequent `NOT_CRAN=true` builds. Correct.

But the configure.ac does NOT invoke `cargo revendor --freeze --compress` for CRAN prep — that is delegated to `just vendor`. This is the right architecture (configure must not modify source files), but means:
- `FORCE_VENDOR=1` in `NOT_CRAN=true` mode runs a bare `cargo revendor` without freeze/compress — only useful for syncing workspace crates into `vendor/` for dev builds
- CRAN prep still requires `just vendor` separately

### `configure.ac` FORCE_VENDOR + `add_vendor_patches` ordering (#246)

The known issue referenced as #246 is about `add_vendor_patches` adding `[patch."https://…"]` entries to Cargo.toml BEFORE `cargo revendor` runs (in the `vendor_miniextendr_local` code path). The configure.ac `cargo-vendor` block does not call `add_vendor_patches` — that's R-side code in `vendor.R`. The template configure.ac invokes `cargo revendor` directly, which means:

1. `vendor.R:vendor_miniextendr_local` copies crates + calls `add_vendor_patches` (writes `[patch."https://…"]` to Cargo.toml)
2. configure.ac `cargo-vendor` block runs `cargo revendor` — which calls `vendor.rs:run_cargo_vendor` — which checks `if !local_pkgs.is_empty() && !ws_original.contains("[patch.crates-io]")` before adding its own patch entries

Since `add_vendor_patches` writes `[patch."https://github.com/…"]` (not `[patch.crates-io]`), the two patch sections don't collide. But the `cargo revendor` invocation from configure.ac has already found the manifest has a `[patch.]` section injected by `add_vendor_patches`, so it skips injecting its own `[patch.crates-io]` for `run_cargo_vendor`. This means the `cargo vendor` subprocess may fail to resolve local crates because it looks for them on crates.io instead of following path deps. The exact interaction depends on whether the git patches already present satisfy the resolver.

The fix path is clear: either configure.ac should not call `add_vendor_patches` (it doesn't today — that's R-side), or `run_cargo_vendor`'s check needs to be smarter (check for `[patch.crates-io]` specifically, not `[patch`).

Looking at vendor.rs:47:
```rust
if !local_pkgs.is_empty() && !ws_original.contains("[patch.crates-io]") {
```

This check already distinguishes: it only skips adding `[patch.crates-io]` if the original manifest **already has** `[patch.crates-io]`. If `add_vendor_patches` added `[patch."https://…"]`, the `[patch.crates-io]` check would still pass and both sections would coexist. This looks safe — but the test coverage for this interaction is absent.

### Template `bootstrap.R` — correct, simple

No issues. Correctly sets `NOT_CRAN=true` default for R CMD build bootstrap runs.

## 6. Template–Tool Coupling

The following implicit contracts exist between templates and cargo-revendor's output. Breaking any of them silently without updating templates creates latent failures.

**6a. Vendor dir name is `vendor/`**

Both `configure.ac` (`VENDOR_OUT="$abs_top_srcdir/vendor"`) and `cargo-config.toml.in` (`directory = "@VENDOR_OUT_CARGO@"`) assume the vendor directory will be named and located at package-root/vendor/. `cargo revendor --output vendor` must produce this. No issue if the flag default stays `vendor`, but a future `--output` rename would silently break the configure.

**6b. `vendor/.cargo-config.toml` naming**

`generate_cargo_config` writes to `vendor/.cargo-config.toml` (note the leading dot, not `.cargo/config.toml`). This is reference-only output; the actual build config is generated by configure.ac from `cargo-config.toml.in`. There is no user-visible coupling issue, but the filename is easily confused with Cargo's actual config file. The README says "Generates `.cargo/config.toml`" (under "What it does", step 8) which is misleading — it generates `vendor/.cargo-config.toml`, not `src/rust/.cargo/config.toml`.

**6c. `inst/vendor.tar.xz` as the canonical CRAN artifact**

Both configure.ac and the pre-commit hook assume `inst/vendor.tar.xz` is the tarball path. `cargo revendor --compress inst/vendor.tar.xz` produces it. The coupling is benign but undocumented as a constraint in the README.

**6d. Lockfile checksums stripped by cargo-revendor**

`strip_lock_checksums` writes a checksum-free Cargo.lock to `vendor/Cargo.lock` and `regenerate_lockfile` copies it back. configure.ac's `cargo-lockfile-compat` block also strips checksums via `sed -i.bak '/^checksum = /d'`. Both tools strip checksums, creating a redundant but non-conflicting operation. If cargo-revendor ever changed to preserve checksums (e.g. for `--no-strip-checksums`), configure.ac would still strip them — the configure.ac acts as a safety net.

**6e. Versioned-dirs layout assumption in `add_vendor_patches`**

`vendor.R:add_vendor_patches` (lines 732–744) correctly handles both versioned and flat layouts:
```r
versioned <- list.files(vendor_dir, pattern = paste0("^", crate, "-[0-9]"), ...)
if (length(versioned) > 0) {
  dir_name <- versioned[[1]]
} else if (dir.exists(file.path(vendor_dir, crate))) {
  dir_name <- crate
```

This is the correct probe-both approach and would survive a layout change. **Good pattern, worth preserving.**

## 7. minirextendr R-side Integration

### `vendor_crates_io` calls `cargo vendor`, not `cargo revendor` — regression hazard

`minirextendr/R/workflow.R:miniextendr_vendor` (the top-level user-facing CRAN prep function) calls `vendor_crates_io()` which shells out to:
```r
# vendor.R:462–464
result <- run_with_logging(
    "cargo",
    args = c("vendor", "--manifest-path", cargo_toml, vendor_dir),
```

This calls **plain `cargo vendor`**, not `cargo revendor`. Plain `cargo vendor` will:
- Skip all workspace/path deps (not copy them into vendor/)
- Not strip test/bench directories
- Not freeze the manifest
- Not handle versioned-dirs consistently

This means the R-side `miniextendr_vendor()` workflow is a **completely separate code path** from what `just vendor` uses. `just vendor` calls `cargo revendor`, but `miniextendr_vendor()` calls `cargo vendor`. The two produce incompatible outputs for any package with workspace path deps.

This is not a conditional divergence or a deprecated code path — it is the primary user-facing vendoring function. Users following the R workflow guide would call `minirextendr::miniextendr_vendor()` which would fail to include workspace crates in the vendor archive, producing a broken CRAN submission.

**This is the most critical correctness finding in this review.**

The R-side workflow should call cargo-revendor. The correct invocation would look like:
```r
result <- run_with_logging(
    "cargo",
    args = c("revendor",
             "--manifest-path", cargo_toml,
             "--output", vendor_dir,
             "--strip-all",
             "--freeze",
             "--compress", tarball,
             "--blank-md",
             "--source-marker",
             "-v"),
    ...
)
```

### `patch_cargo_toml` in vendor.R: fragile regex-based workspace inheritance resolution

`minirextendr/R/vendor.R:patch_cargo_toml` (lines 221–287) uses regex substitution to resolve workspace inheritance fields:
```r
replacements <- list(
    'edition\\.workspace = true' = 'edition = "2024"',
    'version\\.workspace = true' = 'version = "0.1.0"',
```

These hardcode miniextendr-specific values (`"2024"`, `"0.1.0"`, `"MIT"`, etc.). This will silently produce wrong vendor manifests if miniextendr's workspace ever bumps its version or edition. The `resolve_workspace_inheritance` function in `vendor.rs` reads the actual values from the workspace Cargo.toml, avoiding this hardcoding. The R-side code should either call cargo-revendor (which would handle this correctly) or read the actual workspace values dynamically.

### `miniextendr_vendor` in workflow.R: tar invocation missing COPYFILE_DISABLE and --no-xattrs

`workflow.R:miniextendr_vendor` (lines 212–215) creates the tarball with:
```r
tar_output <- system2("tar", c("-cJf", tarball, "-C", staging, "vendor"), ...)
```

No `COPYFILE_DISABLE=1` env var, no `--no-xattrs` flag. On macOS this will embed AppleDouble `._*` files and LIBARCHIVE.xattr PAX headers in the tarball. cargo-revendor's `compress_vendor` (vendor.rs:997–1014) correctly handles this. The R-side `miniextendr_vendor()` will produce tarballs that cause GNU tar warnings on Linux/Windows CRAN machines.

### `vendor_sync` detects source via `.vendor-source` marker

`vendor.R:vendor_sync` reads `.vendor-source` from vendor/ and re-syncs from the recorded local path. cargo-revendor also writes `.vendor-source` when `--source-marker` is passed. If cargo-revendor writes the marker and the R-side `vendor_sync` reads it, they could interoperate — but the R-side `vendor_miniextendr_local` writes the *local path to miniextendr repo*, while cargo-revendor writes whatever `--source-root` or "auto-detected" is. The formats are compatible only by coincidence (both write a path string).

## 8. CRAN Compliance Risks

### `vendor/.cargo-config.toml` — hidden file surviving into tarball

`generate_cargo_config` writes `vendor/.cargo-config.toml` (a dotfile). `strip_vendor_dir` (strip.rs:105–117) in `strip_crate_dir` removes hidden files from individual crate subdirs, but `.cargo-config.toml` is at the top level of `vendor/` — it does not live inside a crate dir and is never stripped. It will be included in `inst/vendor.tar.xz`. CRAN's `--as-cran` check may flag hidden files (starting with `.`) in the package source. The `.vendor-source` marker has the same issue.

Both files are inside `vendor/` (not at the package root), and CRAN checks typically look for non-portable filenames recursively. Whether these dotfiles in vendor/ trigger a NOTE depends on CRAN policy — it's a risk worth noting.

The `.revendor-cache` is also written to `vendor/`. `verify.rs:collect_file_hashes` explicitly skips `.revendor-cache` in the tarball comparison (line 296–299), but it was written to `vendor/` AFTER compression. If the order is correct (compress, then save cache), the tarball would not contain it. Inspecting `main.rs:426–437`:
```
Step 13: compress (if --compress)
Step 14: save_cache
```
Correct — `.revendor-cache` is saved after compression, so it is never in the tarball. The `verify.rs` skip guard is a belt-and-suspenders measure.

### `{"files": {}, "package": null}` vs `{"files":{}}` inconsistency

`vendor.R:vendor_miniextendr_local` (line 199) writes:
```r
writeLines('{"files": {}, "package": null}', checksum_file)
```

But `vendor.rs:clear_checksums` (vendor.rs:584) writes:
```rust
std::fs::write(&cksum, "{\"files\":{}}")?;
```

The R-side includes `"package": null` and uses spaces; the Rust side uses `{"files":{}}` without `"package"`. Cargo accepts both forms, but the inconsistency is unnecessary. The canonical form accepted by cargo is `{"files": {}, "package": null}` or `{"files":{}}` — both are valid TOML-less JSON. Low severity, but worth making consistent.

### Cargo.lock committed with stripped checksums

After `--freeze`, the project's Cargo.lock has no `checksum = ` entries. CRAN requires `Cargo.lock` to be present for reproducible builds. A checksum-free lock is valid for offline builds but would fail `cargo fetch --locked` from a network context. For CRAN purposes (offline-only builds) this is correct. The CLAUDE.md documents this.

## 10. Things That Are Good

**10.1 Versioned-dirs default is architecturally correct** — PR #239's flip to default-on eliminates the entire class of drift bugs where upgrading a dep produces two directories (`mylib/` and `mylib-2.0.0/`) and `cargo vendor`'s choice of which to keep is non-deterministic. The versioned layout is strictly better and should be preserved.

**10.2 `add_vendor_patches` probe-both pattern** — `vendor.R:add_vendor_patches` (lines 732–737) probes for both versioned (`<name>-<version>/`) and flat (`<name>/`) layouts before generating patch entries. This makes the R-side code forward and backward compatible with the layout change. Worth copying this pattern to the test helpers (see S1.2).

**10.3 Cache key includes local crate source trees** — `cache.rs:collect_crate_files` hashes `src/`, `tests/`, `benches/`, `build.rs`, and `Cargo.toml` for each local crate. This catches source-only edits to workspace crates that don't touch Cargo.lock (issue #150). This is strictly better than just hashing the lockfile, which is what naive implementations do. The explicit sort via `BTreeMap` ensures cache stability across OS/filesystem orderings.

**10.4 `scopeguard`-free manifest mutation is safe for normal exit** — The current error path in `run_cargo_vendor` (vendor.rs:68–71) does restore the manifest after the `output.status.success()` check, covering the most common failure mode (cargo vendor exits non-zero). The gap is only for panics/signals.

**10.5 `strip_toml_sections` is line-based but correct** — The text-scanning approach in `strip.rs:132–167` avoids introducing a full TOML parse/rewrite round-trip for stripped sections, which prevents toml_edit from reformatting unrelated sections. Correct output for all test cases including `[[test]]` inside a `[[bench]]` file. The `in_stripped_section` state machine correctly handles back-to-back array-of-tables.

**10.6 `verify_lock_matches_vendor` accepts both flat and versioned layouts** — `verify.rs:105–115` probes `vendor/<name>-<version>/` first, then `vendor/<name>/`. This bidirectional tolerance is exactly right and ensures `--verify` continues to work during any future layout migration.

**10.7 Tarball xattr suppression is thorough** — `compress_vendor` (vendor.rs:1001–1009) dynamically probes for `--no-xattrs` support before passing it, combined with `COPYFILE_DISABLE=1`. This gracefully handles bsdtar versions that don't support `--no-xattrs`. The same result could be achieved with `COPYFILE_DISABLE` alone on modern macOS, but having `--no-xattrs` as belt-and-suspenders is correct.

**10.8 `pathdiff` produces relative paths without depending on a crate** — The hand-rolled `pathdiff` in vendor.rs:843–874 avoids adding a dependency on the `pathdiff` crate. For a standalone binary that must stay lean, this is appropriate.

**10.9 `BTreeSet` for patched_names in freeze_manifest ensures determinism** — vendor.rs:722–743 collects crate names from `[patch.*]` sections into a `BTreeSet`, which (combined with the alphabetical `BTreeMap` for the new `patch_table`) produces deterministically ordered `[patch.crates-io]` entries. This was the fix for PR #205 and it is correct.
