# Investigation: rpkg/src/rust/.cargo/config.toml deletion fate

Item 7 of `plans/lockfile-mode-unification.md`.

## What was attempted

Investigated why `rpkg/src/rust/.cargo/config.toml` has been observed to disappear
after certain dev operations. This file is generated by `bash ./configure` and
carries the `[patch."https://github.com/A2-ai/miniextendr"]` block that redirects
framework crates to local monorepo siblings in source-mode dev. When it vanishes,
`cargo build` fetches from `git+url#<sha>` instead of local siblings.

Ran each hypothesis against the code, simulated the key scenario by creating a
fake `inst/vendor.tar.xz` and running configure, then traced through the R install
internals to confirm which paths operate on the source tree vs. a temp copy.

## What went wrong (the real cause)

The deletion is **real and reproducible**. There is one concrete trigger path:

### The stale-tarball trap-failure scenario

1. Developer runs `just r-cmd-build` or `just r-cmd-check`.
2. Either recipe depends on `just vendor`, which creates `rpkg/inst/vendor.tar.xz`.
3. The recipe sets `trap 'rm -f rpkg/inst/vendor.tar.xz' EXIT` to clean up.
4. If the process is interrupted (Ctrl-C, SIGKILL, OOM) **before the trap fires**,
   `inst/vendor.tar.xz` is left in the source tree.
5. Developer later runs `just rcmdinstall` (or `R CMD INSTALL rpkg`).
6. `just rcmdinstall` depends on `just configure` → configure sees
   `inst/vendor.tar.xz` → sets `IS_TARBALL_INSTALL=true`.
7. `R CMD INSTALL rpkg` calls `./configure` and `make` **in the source directory**
   (confirmed from `tools:::.install_packages` source: when the argument is a
   directory, `allpkgs <- c(allpkgs, fullpath(pkg))`, then `setwd(pkg)`; there is
   no temp copy for source-directory installs).
8. `make` runs the cleanup block in `Makevars.in` (lines 54–63, gated on
   `IS_TARBALL_INSTALL=true`):
   ```makefile
   if [ -d "$(ABS_RPKG_SRCDIR)/rust/.cargo" ]; then rm -rf "$(ABS_RPKG_SRCDIR)/rust/.cargo"; fi
   ```
   `ABS_RPKG_SRCDIR` is an absolute path to the **source** `rpkg/src/`, not a
   temp dir — so `src/rust/.cargo/` is deleted from the working tree.
9. `src/rust/.cargo/config.toml` is gone. The next `cargo build` resolves the
   three workspace crates via the bare `git+url#<sha>`, ignoring local edits.

### When each recipe is safe

| Command | vendor.tar.xz present? | Operates on source dir? | Can delete config.toml? |
|---|---|---|---|
| `just rcmdinstall` / `R CMD INSTALL rpkg` | Only if stale from prior failed run | Yes — source dir | **YES, if tarball present** |
| `just r-cmd-build` | Yes (via `just vendor` dep) | No — build copies to temp | No (tarball stripped by trap; if trap fails, next install triggers it) |
| `just r-cmd-check` | Yes (via `just vendor` dep) | No — R CMD check temp dir | No (same as above) |
| `just devtools-install` | No | Yes — source dir | No (tarball absent → source mode) |
| `just devtools-document` | No | No compilation | No |
| `just devtools-test` | No | No | No |
| `just devtools-check` | No | No — devtools::check builds tarball in temp | No |

Key finding: **`just devtools-check` is safe** because `devtools::check()` runs
`R CMD build` to produce a temp tarball and `R CMD check` on that tarball. Neither
operates on `inst/vendor.tar.xz` and neither touches the source `.cargo/`.

**`R CMD INSTALL` on a tarball** (e.g., `R CMD INSTALL miniextendr_0.1.0.tar.gz`)
is also safe: R extracts to `tmpdir` first (`allpkgs <- c(allpkgs, file.path(tmpdir, pkgname))`),
so the cleanup block fires on the temp dir, not the source tree.

## Root cause summary

Two independent facts combine to cause the problem:

1. **`just r-cmd-build` / `r-cmd-check` leave `inst/vendor.tar.xz` behind when
   interrupted.** The trap `rm -f rpkg/inst/vendor.tar.xz` is correct but only
   fires on clean EXIT/ERR, not SIGKILL.

2. **`R CMD INSTALL <sourcedir>` operates in-place.** The cleanup block in
   `Makevars.in` is intentionally designed for tarball-installs (where `ABS_RPKG_SRCDIR`
   points to a temp dir). But when `inst/vendor.tar.xz` is accidentally present in
   the source tree and a developer runs `R CMD INSTALL rpkg`, configure sets
   `IS_TARBALL_INSTALL=true` and `ABS_RPKG_SRCDIR` points to the actual source path.

## Hypotheses ruled out

1. **`R CMD INSTALL` clean-up step** — ruled out for tarball-installs (temp dir);
   confirmed real for source-dir installs when `inst/vendor.tar.xz` is present.
2. **`pkgbuild::compile_dll()` cleaning cargo target dir** — ruled out. `.cargo/`
   is not the target dir; pkgbuild touches `target/` only.
3. **`R CMD INSTALL --no-clean-on-error`** — not relevant; the deletion is in the
   success path of `make all`, not a cleanup-on-error hook.
4. **`tools/*.R` side-effect** — ruled out. `detect-features.R` only emits feature
   names to stdout; no file mutations.
5. **Cross-worktree illusion** — tested directly in this worktree. The deletion
   is reproducible by simulating the stale-tarball condition.
6. **`R CMD build` touching source `.cargo/config.toml`** — ruled out. `R CMD build`
   extracts source to a temp tree before building the tarball.
7. **`cargo clean` inadvertently called** — ruled out. `cargo clean` only removes
   `target/`; `.cargo/` is separate.

## Fix / mitigation (applied in this PR)

### 1. Stale-tarball guard in `configure.ac`

Added a prominent warning in `configure.ac` near the `cargo-config` block when
`IS_TARBALL_INSTALL=true` but the current directory looks like a source-mode dev
tree (presence of `Cargo.lock` in VCS, absence of `vendor/` in expected locations, etc.).

### 2. `minirextendr_doctor()` check for stale tarball

`minirextendr/R/doctor.R`: added a check that warns when `inst/vendor.tar.xz`
exists in the project (outside CI / release context) and recommends removing it
with `rm rpkg/inst/vendor.tar.xz && just configure`.

### 3. `docs/CRAN_COMPATIBILITY.md` note

Added a one-line note in the "Constraints" section warning that a leftover
`inst/vendor.tar.xz` in a source-mode working tree will flip configure to tarball
mode and trigger the `src/rust/.cargo/` cleanup on next install.

### 4. CLAUDE.md Common Issues entry

Added entry: "Missing `.cargo/config.toml`" pointing to stale `inst/vendor.tar.xz`
as the cause.

## Symptom mapping

User sees, after `just rcmdinstall` following an interrupted `r-cmd-build`/`r-cmd-check`:
- Rust build succeeds but resolves framework crates from `git+https://...#<sha>` instead of local workspace siblings.
- Local edits to `miniextendr-api` / `miniextendr-macros` / `miniextendr-lint` are silently ignored.
- Next `just rcmdinstall` in source mode also fails to see local edits (the file is gone until `just configure` restores it).
- `just configure` re-generates the file, but if `inst/vendor.tar.xz` is still present, configure regenerates it in **tarball mode** (wrong config) and the next `R CMD INSTALL` deletes it again.
- To escape: `rm rpkg/inst/vendor.tar.xz && just configure`.
