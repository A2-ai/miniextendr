# `miniextendr` example R package

Example R package that exercises the `miniextendr` Rust crates. It uses a Rust
backend with autoconf-based configuration and supports both monorepo
development and standalone/offline installation.

## Build system overview

This package embeds a Rust crate under `src/rust/` and builds it during
`R CMD INSTALL`. The key stages are:

1. `bootstrap.R`
   - Runs `configure` and stops.
   - Exists so monorepos can keep the R package in a subdirectory while still
     using R's package build flow.

2. `configure`
   - Generated from `configure.ac` via `autoconf`.
   - Generates `src/Makevars`, `src/rust/Cargo.toml`, and
     `src/rust/.cargo/config.toml`.
   - In dev mode it can sync local workspace crates into `vendor/`.
   - In CRAN/offline mode it expects vendored sources or `inst/vendor.tar.xz`
     to already exist.

3. `src/Makevars`
   - Builds the Rust static library with `cargo build`.
   - Builds a temporary cdylib solely to write `R/miniextendr-wrappers.R`.
   - Unpacks `inst/vendor.tar.xz` during offline/CRAN installs when `vendor/`
     is absent.
   - Does not run `cargo vendor`; vendoring is handled before build time.

## Templates

The `minirextendr` templates are kept in sync with this package. See
`../minirextendr/inst/templates/README.md` and run `just templates-check` from
the repo root to verify template drift.

## Files that must be committed

The following generated files must be committed to version control:

- `configure` - generated from `configure.ac` by `autoconf`
- `R/miniextendr-wrappers.R` - generated by the wrapper-generation entrypoint
- `config.guess` and `config.sub` - copied in for portability

These stay committed because:

- users should not need autoconf installed
- `R CMD build` expects the wrapper file to exist
- CRAN requires portable `configure` scripts and vendored build inputs

## Developer guide

Typical workflows from the repo root:

```sh
just configure        # generate Makevars + .cargo/config.toml (auto-detects mode)
just devtools-load    # compile Rust, regenerate wrappers, load the package
just devtools-test    # run the testthat suite
just vendor           # CRAN release prep: create inst/vendor.tar.xz
just r-cmd-build      # build tarball (depends on `just vendor`)
just r-cmd-check      # build tarball and run R CMD check
```

### Regenerating `configure`

When you modify `configure.ac`, regenerate `configure`:

```sh
cd rpkg
autoconf
```

### Regenerating R wrappers

When you change the exported Rust API (add/remove/rename exported functions or
methods), regenerate the wrappers:

```sh
just devtools-document
```

That rebuilds the package, rewrites `R/miniextendr-wrappers.R`, and refreshes
`NAMESPACE` / roxygen output.

## Source vs tarball install

- **Source install** (default for `R CMD INSTALL .`, `devtools::install`,
  `install_github`): run `bash ./configure` (or `just configure`). Cargo
  resolves miniextendr deps from monorepo siblings when configure detects
  them, otherwise from the git URL declared in `Cargo.toml`. No vendoring.
- **Tarball install** (CRAN, `R CMD INSTALL <built-tarball>`): triggered
  automatically when `inst/vendor.tar.xz` is present. configure unpacks it
  and writes a vendored cargo source-replacement, builds offline. The single
  signal is the file's existence — no env var to set. Maintainers run
  `just vendor` to produce it, `R CMD build` packages it.

See [docs/CRAN_COMPATIBILITY.md](../docs/CRAN_COMPATIBILITY.md) for the full table.

## Publishing to CRAN

This package is structured to be CRAN-compatible if you follow these rules:

- ensure all Rust dependencies are vendored and included in the source tarball
- keep `configure`, `config.guess`, and `config.sub` committed and up to date
- regenerate `R/miniextendr-wrappers.R` before release
- avoid non-API R symbols unless you are prepared to handle CRAN checks
- run `R CMD check` on the built tarball

## Maintainer

- Update `config.guess` and `config.sub` periodically from GNU config:

```text
https://cgit.git.savannah.gnu.org/cgit/config.git/tree/
```

- When `configure.ac` changes, regenerate `configure` and commit it.
- Keep `configure.win` and `configure.ucrt` wrappers in sync with `configure`.
- If the Rust API changes, regenerate wrappers and review exported symbols.
- Rebuild the tarball and run `R CMD check` before release.
