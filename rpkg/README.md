# Rust-based R package `rpkg`

Example R package that exercises the `miniextendr` Rust crates. It uses a Rust
backend with autoconf-based configuration and supports both monorepo development
and standalone installation.

## Build system overview

This package embeds a Rust crate under `src/rust/` and builds it during
`R CMD INSTALL`. The key stages are:

1) `bootstrap.R` (pre-build, configured by `Config/build/bootstrap: TRUE`)
   - Only runs `rpkg/configure` and stops. It is intentionally simple.
   - Exists to support monorepos where the R package lives alongside the Rust
     crate it exposes to R (i.e., the package is a subdirectory of the Rust
     workspace).

2) `configure`
   - Generated from `configure.ac` via `autoconf`.
   - Sets `NOT_CRAN=true` during development.
   - Generates `src/Makevars`, `src/rust/Cargo.toml`, and
     `src/rust/.cargo/config.toml`.

3) `Makevars`
   - Runs `cargo vendor` when `NOT_CRAN=true`.
   - Builds Rust code with `cargo build`.
   - Generates R wrappers with `cargo run --bin document`.
   - Links the shared library.

## Files that must be committed

The following generated files **must** be committed to version control:

- `configure` – generated from `configure.ac` by `autoconf`.
- `R/miniextendr_wrappers.R` – generated by the `document` binary.
- `config.guess` and `config.sub` – copied in and committed for portability.

These are committed because:
- Users should not need autoconf installed.
- `R CMD build` expects R wrapper files to exist.
- CRAN requires portable `configure` scripts.

## Developer guide

### Regenerating configure

When you modify `configure.ac`, regenerate `configure`:

```sh
cd rpkg && autoconf
```

### Regenerating R wrappers

When you modify Rust code that changes the R API (add/remove/rename exported
functions), regenerate the wrappers:

```sh
just devtools-load  # or devtools::load_all("rpkg")
```

This runs cargo build and the document binary, then copies the generated
`src/miniextendr_wrappers.R` to `R/miniextendr_wrappers.R`.

## CRAN vs development builds

- **CRAN builds**: `NOT_CRAN` is not "true", so vendoring is skipped. The
  tarball must already contain all vendored dependencies in `src/vendor/`.

- **Development builds**: `NOT_CRAN=true` (set by devtools or configure), so
  `cargo vendor` runs to refresh dependencies.

## Publishing to CRAN

This package is structured to be CRAN-compatible provided you follow these
rules:

- Ensure all Rust dependencies are vendored in `src/vendor/` and included in
  the source tarball.
- Do not enable non-API R symbols unless you are prepared to handle CRAN checks
  (see `NONAPI.md`).
- Keep `configure`, `config.guess`, and `config.sub` committed and up to date.
- Regenerate `R/miniextendr_wrappers.R` before release.
- Run `R CMD check` on the built tarball.

## Maintainer

- Update `config.guess` and `config.sub` periodically from GNU config:

```text
https://cgit.git.savannah.gnu.org/cgit/config.git/tree/
```

- After updating, ensure the files are executable where required.
- When `configure.ac` changes, regenerate `configure` and commit it.
- Keep `configure.win` and `configure.ucrt` wrappers in sync if the base
  `configure` changes.
- If the Rust API changes, regenerate wrappers and review exported symbols.
- Keep `DESCRIPTION` fields current (`Title`, `Description`, `Authors@R`).
- Rebuild the tarball and run `R CMD check` before release.
