# #448 — Release-workflow platform gotchas (AlmaLinux 8 + macOS arm64)

Closes #448.

## Background (one paragraph for the implementer)

A user dry-ran an `r-release.yml`-shaped workflow against a downstream
miniextendr-consumer in `almalinux:8` containers + macOS arm64 matrix and
hit four reproducible gotchas:

1. AlmaLinux 8 minimal defaults to `LANG=C` → `package_init`'s UTF-8
   assertion (`miniextendr-api/src/encoding.rs:88`) fires:
   `miniextendr requires a UTF-8 locale`.
2. Setting `LANG=C.UTF-8` workflow-wide kills macOS — macOS rejects
   `C.UTF-8` (glibc-only), `setlocale(3)` falls back to `C`, same UTF-8
   assertion fires.
3. AlmaLinux minimal ships without `git`; cargo's libgit2 path can't read
   git's credential helper for private deps.
4. macOS runners ship `git` but still need auth + `CARGO_NET_GIT_FETCH_WITH_CLI=true`.

These aren't miniextendr bugs — they're platform behaviour every consumer
will rediscover. miniextendr itself doesn't ship `r-release.yml`, but it
ships `minirextendr` (the scaffolding helper) and project docs that
downstream packages copy from. This plan lands the fixes in two durable
forms: a docs page in `docs/` and a workflow template in
`minirextendr/inst/templates/`.

## Out-of-scope (explicit non-goals)

- Don't change `package_init`'s UTF-8 assertion. The check is correct;
  the platforms are wrong (or differ).
- Don't add a workflow to *miniextendr's own* `.github/workflows/`. This
  repo doesn't release via that path.
- Don't ship a generic `R-CMD-check.yml` template. Scope is `r-release.yml`
  only.

## Plan (flat, priority order)

### 1. Write `docs/RELEASE_WORKFLOW.md`

Single canonical doc. Structure:

- **Why this doc exists** — `R-CMD-check.yaml` doesn't exercise AlmaLinux
  8 / macOS arm64; release workflows do; consumers hit the same four
  gotchas. miniextendr's own CI (`ci.yml`) doesn't release, so the
  expertise lives in prose + a template, not in this repo's workflows.
- **Gotcha 1: AlmaLinux 8 minimal defaults to `C` locale.** Fix:
  `LANG=C.UTF-8` at the *job* env, scoped to container jobs only. Use
  `LC_ALL=C.UTF-8` if `LANG` alone isn't honoured. Alternative: install
  `glibc-langpack-en` and use `en_US.UTF-8`.
- **Gotcha 2: macOS rejects `C.UTF-8`.** Don't set `LANG`/`LC_ALL`
  workflow-level. Scope it to container jobs. macOS runners are UTF-8 by
  default and break if you force `C.UTF-8`.
- **Gotcha 3: AlmaLinux minimal lacks `git`; cargo libgit2 can't auth
  private deps.** Fix:
  - `dnf install -y git gh` (gh repo: `cli.github.com/packages/rpm/gh-cli.repo`).
  - `gh auth setup-git` (or `git config --global url."https://${PAT}@github.com/".insteadOf "https://github.com/"`).
  - `CARGO_NET_GIT_FETCH_WITH_CLI=true`.
- **Gotcha 4: macOS needs auth + git-fetch-with-cli, no extra packages.**
  `gh auth setup-git` + `CARGO_NET_GIT_FETCH_WITH_CLI=true`.
- **Why `package_init`'s UTF-8 check exists** — three-paragraph explainer
  pointing at `miniextendr-api/src/encoding.rs:88`. Reassures the
  downstream maintainer that this isn't a bug to file upstream.
- **Reference template** — link to `minirextendr/inst/templates/r-release.yml`
  and `minirextendr::use_release_workflow()`.

After writing, run `just site-docs` so `site/content/manual/release-workflow.md`
is regenerated and `site/data/plans.json` is up-to-date. Commit both
together per the CLAUDE.md "auto-generated" note.

### 2. Add `minirextendr/inst/templates/r-release.yml`

A minimal release workflow that demonstrates the four fixes baked in.
**Don't try to be exhaustive** — the goal is "drop-in baseline that
loads on AlmaLinux + macOS arm64", not a full CRAN release pipeline.

Skeleton (the implementer should round out, but must include the env
patterns documented in §1):

```yaml
name: r-release

on:
  workflow_dispatch:
    inputs:
      dry_run:
        description: "Skip uploads"
        type: boolean
        default: true
  push:
    tags: ["v*"]

env:
  CARGO_NET_GIT_FETCH_WITH_CLI: "true"

jobs:
  build-primary:
    runs-on: ubuntu-latest
    container: almalinux:8
    env:
      LANG: C.UTF-8
      LC_ALL: C.UTF-8
    steps:
      - name: Install build tools (AlmaLinux 8 minimal)
        run: |
          dnf install -y --setopt=install_weak_deps=False \
            git gh dnf-plugins-core which procps-ng glibc-langpack-en
      - uses: actions/checkout@v4
      - name: Configure git auth for cargo private deps
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: gh auth setup-git
      # ... R setup, cargo, build, R CMD check ...

  build-macos:
    strategy:
      matrix:
        r-version: ['release', 'oldrel-1']
    runs-on: macos-14   # arm64
    steps:
      - uses: actions/checkout@v4
      - name: Configure git auth for cargo private deps
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: gh auth setup-git
      - uses: r-lib/actions/setup-r@v2
        with:
          r-version: ${{ matrix.r-version }}
      # ... cargo, build, R CMD check ...
```

Add a banner comment at the top of the template:

```
# This template encodes four platform gotchas surfaced by issue #448:
#   1. LANG=C.UTF-8 scoped to container jobs (AlmaLinux 8 has no UTF-8 locale)
#   2. NOT set workflow-level (macOS rejects C.UTF-8)
#   3. AlmaLinux minimal needs `dnf install git gh` + `gh auth setup-git`
#   4. CARGO_NET_GIT_FETCH_WITH_CLI=true for cargo + private git deps on both platforms
# See docs/RELEASE_WORKFLOW.md for the rationale.
```

If a `{{package_name}}` placeholder is needed for templating, follow the
convention used by other templates in `minirextendr/inst/templates/`
(check `monorepo/Cargo.toml.tmpl` for the substitution pattern). If no
substitution is needed, ship the file as `r-release.yml` (no `.tmpl`).

## Common pitfalls

- **Don't put `LANG: C.UTF-8` at workflow-level `env:`.** That breaks
  macOS as documented in #448 §2. Job-level `env:` only.
- **`gh auth setup-git` needs a token in env**: set `GH_TOKEN:
  ${{ secrets.GITHUB_TOKEN }}` for that step. Without it, `gh` errors
  with "no token".
- **Template path discovery from R**: use
  `system.file("templates", "r-release.yml", package = "minirextendr")`
  — don't hard-code paths.
- **Zola frontmatter**: `docs/RELEASE_WORKFLOW.md` becomes
  `site/content/manual/release-workflow.md` via the conversion script.
  `scripts/docs-to-site.sh` derives frontmatter from the first `# Heading`
  and first paragraph — pick the heading carefully (it becomes the page
  title) and lead with a one-sentence summary.
- **`Config/build/copy-method` differences**: not relevant here; the
  release workflow runs against the in-tree source, no bootstrap path
  involvement.
