# Vendoring Strategy

miniextendr uses a **single workspace-level vendor directory** for development,
with R package-specific vendoring for CRAN builds.

## Strategy: Workspace ./vendor/ + R Package rpkg/src/vendor/

### 1. Workspace Development (./vendor/)

**Purpose:** Offline development for all workspace crates

**Created by:**
```sh
just vendor  # Syncs all workspace + bench Cargo.toml files
```

**Used by:**
- miniextendr-api (development)
- miniextendr-macros (development)
- miniextendr-bench (benchmarks)
- miniextendr-engine (development)

**Configuration:** Workspace-level .cargo/config.toml (if needed)

**When to run:** After adding/updating dependencies in workspace crates

### 2. R Package Vendoring (rpkg/src/vendor/)

**Purpose:** Self-contained R package for CRAN submission

**Created by:**
```sh
just configure  # Runs configure.ac which:
                # 1. rsync miniextendr-api/ -> rpkg/src/vendor/miniextendr-api/
                # 2. rsync miniextendr-macros/ -> rpkg/src/vendor/miniextendr-macros/
                # 3. cargo vendor crates.io deps -> rpkg/src/vendor/
```

**Used by:**
- R package builds (R CMD INSTALL)
- R CMD check (CRAN validation)

**Configuration:** rpkg/src/rust/.cargo/config.toml (generated by configure)

**When to run:**
- Before building R package
- After modifying miniextendr-api or miniextendr-macros
- Before R CMD check

### 3. What's NOT Vendored

**miniextendr-engine:**
- NOT vendored into rpkg (documented in configure.ac)
- Only used by standalone binaries/benchmarks that embed R
- Not needed for R packages (they run inside R)

## Summary

**Development workflow:**
```sh
# 1. Update workspace deps (if needed)
just vendor

# 2. Work on code...

# 3. Build R package
just configure  # Syncs to rpkg/src/vendor/
just rcmdinstall
```

**CRAN submission:**
- Only rpkg/src/vendor/ is included in tarball
- Workspace ./vendor/ is .gitignored
- miniextendr-bench/vendor/ removed (uses workspace vendor)
